# 迁移指南

当你要把散落的权限判断迁入 permission-core 时，使用本页。

目标是逐步替换旧判断，并在每一步都保留可验证的拒绝路径与回滚能力。

## 建议迁移路径

1. 分开盘点接口权限、集合权限、行级条件和字段权限。
2. 把接口守卫统一成 `<METHOD>:<path>`，使用命中路由模板而不是实际 ID URL。
3. 把数据权限统一成 `db:<collection>[:<field>]`，行级条件放在结构化 `where` 中。
4. 先跑通一个角色、一个用户绑定、一个允许和一个拒绝，再扩大范围。
5. 运行路径清楚后再接入管理 API。
6. 每条规则或用户绑定变更都通过公开 manager，或在绕过时显式处理缓存失效。

## 迁移期间不要做什么

- 不要把真实记录 ID 编进路由资源。
- 不要把认证职责迁入 permission-core。
- 不要把 `getResources()` 当作服务端最终鉴权结果。
- 支付或账本写操作不要未经审查就使用宽泛通配符。

## 版本升级检查

升级已经接入的 permission-core 时：

1. 阅读变更日志，确认 `action + resource`、请求侧 `write` 或公开子路径是否变化。
2. 使用打包产物重新运行当前快速开始和维护中的示例，不要只验证源码导入。
3. 重新验证自定义存储、缓存所有权、自定义资源 scheme 和框架路由映射。
4. 启用菜单或 scope API 时，同时迁移核心存储与菜单存储，并证明租户分区仍然精确。
5. 先部署兼容旧格式的读取端，再写入新的持久化结构；数据变化后使权限缓存失效。
6. 保留核心角色、规则、绑定，以及菜单节点、API 绑定、revision 和审计记录的回滚副本。
